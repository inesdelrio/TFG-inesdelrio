# RF08 - Edicion y eliminacion de eventos

## Estado del requisito

Cerrado y validado en entorno local.

## Objetivo

Permitir que la entidad autora de un evento publicado pueda editar sus datos o retirarlo del sistema, manteniendo control de permisos para impedir modificaciones por parte de otras entidades.

## Alcance funcional

- Existe pantalla de edicion en `GET /eventos/:eventId/editar`.
- La actualizacion se procesa en `POST /eventos/:eventId`.
- La eliminacion se procesa en `POST /eventos/:eventId/eliminar`.
- Solo la entidad propietaria puede editar o borrar sus eventos.
- El listado publico muestra accesos de gestion para la entidad autora autenticada.

## Implementacion realizada

### 1. Rutas y control de acceso

- Se han añadido las rutas protegidas de edicion y eliminacion en `src/routes/events/index.js`.
- Las operaciones requieren sesion activa y rol `ENTIDAD`.
- El servicio `getOwnedEventById` comprueba que el evento pertenece a la entidad autenticada.

### 2. Edicion de evento

- Se ha creado `src/controllers/events/event-management.controller.js`.
- Se ha creado la vista `src/views/pages/events/edit.ejs`.
- El formulario de edicion precarga los datos actuales del evento.
- Se reutiliza la validacion de `RF07` para mantener reglas de fecha, hora, plazas y contenido.

### 3. Eliminacion de evento

- Se ha creado el servicio `src/services/events/delete-owned-event.service.js`.
- La eliminacion borra el evento de base de datos solo si la entidad autora coincide.
- Tras eliminar, el usuario vuelve al listado general con confirmacion.

### 4. Persistencia y permisos

- Se ha creado `src/services/events/update-owned-event.service.js`.
- Se ha creado `src/services/events/get-owned-event-by-id.service.js`.
- Ambos servicios aplican control de propiedad del evento.
- Si la entidad no es la autora, la operacion se bloquea.

### 5. Interfaz

- El listado publico de eventos muestra acciones `Editar` y `Eliminar` solo para la entidad propietaria autenticada.
- La vista de edicion incluye acceso de vuelta al listado y accion de borrado.

### 6. Tests unitarios

- Se ha añadido un test unitario de actualizacion de evento propio.
- Se ha añadido un test unitario de eliminacion de evento propio.
- Se ha añadido un test unitario de denegacion de eliminacion cuando el evento no pertenece a la entidad.

## Migraciones y base de datos

- `RF08` no requiere cambios de esquema en Prisma.
- No ha sido necesaria ninguna migracion nueva porque reutiliza el modelo `Event` creado en `RF07`.

## Validacion realizada

- `npm run prisma:validate` ejecutado correctamente.
- `npm test` ejecutado correctamente.
- `GET /eventos/:eventId/editar` responde `200` para la entidad autora.
- `POST /eventos/:eventId` responde `302` a `/eventos?updated=1`.
- `POST /eventos/:eventId/eliminar` responde `302` a `/eventos?deleted=1`.
- La entidad no propietaria queda bloqueada para editar o borrar.

## Comprobacion manual

1. Ejecutar `npm run dev`.
2. Iniciar sesion con una cuenta de entidad que tenga un evento publicado.
3. Abrir `http://localhost:3000/eventos`.
4. Pulsar `Editar` sobre un evento propio.
5. Modificar datos y guardar.
6. Verificar la redireccion a `/eventos?updated=1`.
7. Volver al listado y comprobar los cambios.
8. Probar la accion `Eliminar` y comprobar la desaparicion del evento.

## Evidencia de cierre

- Flujo HTTP validado en local con una entidad temporal autora de un evento.
- El evento de prueba pudo editarse correctamente y luego eliminarse.
- El control de propiedad queda cubierto en servicios y tests unitarios.
