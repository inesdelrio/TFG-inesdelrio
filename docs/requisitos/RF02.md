# RF02 - Inicio y cierre de sesion

## Estado del requisito

Cerrado definitivamente y validado en entorno local.

## Objetivo

Permitir que un usuario existente inicie sesion con email y contraseña, quede autenticado en servidor mediante `express-session`, acceda a su area privada segun el rol y pueda cerrar sesion correctamente.

## Implementacion realizada

### 1. Sesion en Express

- Se ha configurado `express-session` en `src/app.js`.
- La sesion usa `SESSION_SECRET` desde `.env`.
- La cookie de sesion se crea con `httpOnly`, `sameSite=lax` y expiracion de 8 horas.
- El usuario autenticado se expone en `req.currentUser` y `res.locals.currentUser`.

### 2. Login y logout

- Se ha creado la ruta `GET /login`.
- Se ha creado la ruta `POST /login`.
- Se ha creado la ruta `POST /logout`.
- El login consulta el usuario por `email` en PostgreSQL.
- La contraseña introducida se compara con `bcrypt`.
- Si el login es correcto, se guarda el usuario en sesion.
- Si el login falla, se muestra error controlado en la interfaz.

### 3. Redireccion por rol

- Se ha creado la utilidad `src/utils/auth-redirect.js`.
- El sistema redirige a:
  - `VOLUNTARIO` -> `/voluntariado/area`
  - `ENTIDAD` -> `/entidad/area`
  - `ADMIN` -> `/admin/area`

### 4. Area autenticada y proteccion

- Se ha creado el middleware `requireAuth`.
- Se ha creado el middleware `requireRole`.
- Se ha creado el area inicial del voluntario en `src/views/pages/volunteers/area.ejs`.
- Se han dejado areas base para entidad y administracion para soportar la redireccion por rol.
- Las rutas privadas redirigen a `/login` si no existe sesion activa.

### 5. Interfaz

- Se ha creado la vista `src/views/pages/auth/login.ejs`.
- Se ha añadido la accion de cierre de sesion al menu principal.
- La cabecera ya adapta su contenido segun exista o no usuario autenticado.
- El flujo de registro ahora informa de que el usuario ya puede iniciar sesion.

### 6. Tests unitarios

- Se ha añadido test unitario de login valido en el servicio de autenticacion.
- Se ha añadido test unitario de login invalido.
- Se ha añadido test unitario del cierre de sesion.
- Se ha añadido test unitario de validacion de datos invalidos en login.

## Validacion realizada

- `npm test` ejecutado correctamente.
- `npm run prisma:validate` ejecutado correctamente.
- `POST /login` con credenciales validas devuelve `302` a `/voluntariado/area`.
- `GET /voluntariado/area` con sesion activa responde `200`.
- `POST /logout` devuelve `302` a `/login`.

## Comprobacion manual

1. Ejecutar `npm run dev`.
2. Abrir `http://localhost:3000/login`.
3. Iniciar sesion con un usuario existente.
4. Verificar la redireccion al area privada correspondiente.
5. Refrescar la pagina y comprobar que la sesion sigue activa.
6. Pulsar `Cerrar sesion`.
7. Verificar la redireccion a `/login`.
8. Intentar abrir `/voluntariado/area` sin sesion y comprobar que redirige a `/login`.
