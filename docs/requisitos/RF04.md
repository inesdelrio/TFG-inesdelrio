# RF04 - Verificacion de entidades

## Estado del requisito

Cerrado definitivamente y validado en entorno local.

## Objetivo

Permitir que un administrador revise solicitudes de entidad, cambie su estado de validacion y deje trazabilidad administrativa del cambio.

## Implementacion realizada

### 1. Panel base de administracion

- Se ha creado el router `src/routes/admin/index.js`.
- La ruta `GET /admin/area` muestra el listado de entidades pendientes.
- La ruta `GET /admin/entidades/:entityId` muestra el detalle de la solicitud.
- Las rutas administrativas requieren sesion activa y rol `ADMIN`.

### 2. Cambio de estado de entidad

- Se ha creado la ruta `POST /admin/entidades/:entityId/estado`.
- El controlador administrativo procesa acciones de `VERIFICADA`, `RECHAZADA` y `SUSPENDIDA`.
- El servicio `updateEntityValidationStatus` actualiza el estado en base de datos.
- El servicio ya bloquea transiciones invalidas y cambios al mismo estado.

### 3. Trazabilidad administrativa

- Se ha añadido el modelo `AdminActionLog` en Prisma.
- Cada cambio de estado genera un log con:
  - accion administrativa
  - estado anterior
  - estado nuevo
  - notas
  - administrador responsable
  - entidad afectada

### 4. Restriccion para publicacion de eventos

- Se ha creado `src/services/entities/entity-publication-access.service.js`.
- Este servicio lanza error si la entidad no esta en estado `VERIFICADA`.
- Queda preparado para usarse en `RF07` cuando se implemente la publicacion de eventos.

### 5. Interfaz

- Se ha creado el listado administrativo en `src/views/pages/admin/entities-list.ejs`.
- Se ha creado la vista de detalle y moderacion en `src/views/pages/admin/entity-detail.ejs`.
- El detalle muestra datos de la entidad, solicitante, estado actual, formulario de cambio y log historico.

### 6. Bootstrap de admin local

- Se ha añadido el script `npm run seed:admin`.
- El script crea o actualiza un administrador local para pruebas.
- El email y la contraseña se leen desde `ADMIN_EMAIL` y `ADMIN_PASSWORD` en `.env`.
- Si falta alguna de esas variables, el script falla con un mensaje claro y no crea credenciales por defecto.

- La migracion crea la tabla `AdminActionLog` y sus claves foraneas con `User` y `Entity`.

## Validacion realizada

- `npm run prisma:validate` ejecutado correctamente.
- `npm test` ejecutado correctamente.
- `npm run seed:admin` ejecutado correctamente con `ADMIN_EMAIL` y `ADMIN_PASSWORD` configuradas en `.env`.
- `POST /login` con la cuenta definida en `ADMIN_EMAIL` devuelve `302` a `/admin/area`.
- `GET /admin/area` responde `200`.
- `GET /admin/entidades/1` responde `200`.
- `POST /admin/entidades/1/estado` responde `302` a `/admin/area?updated=1`.
- En base de datos, la entidad `Fundacion Horizonte` queda en estado `VERIFICADA`.
- En base de datos, existe log administrativo asociado a la revision.

## Comprobacion manual

1. Ejecutar `npm run seed:admin`.
2. Ejecutar `npm run dev`.
3. Iniciar sesion con el email y la contraseña configurados en `ADMIN_EMAIL` y `ADMIN_PASSWORD`.
4. Abrir `http://localhost:3000/admin/area`.
5. Entrar al detalle de una entidad.
6. Seleccionar `Verificar`, `Rechazar` o `Suspender`.
7. Guardar el cambio y comprobar la redireccion al listado.
8. Revisar el historial de acciones en el detalle de la entidad.
