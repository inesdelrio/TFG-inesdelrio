# RF04 - Verificacion de entidades

## Estado del requisito

Cerrado definitivamente y validado en entorno local.

## Objetivo

Permitir que un administrador revise solicitudes de entidad, cambie su estado de validacion y deje trazabilidad administrativa del cambio.

## Implementacion realizada

### 1. Panel base de administracion

- Se ha creado el router `src/routes/admin/index.js`.
- La ruta `GET /admin/area` muestra el panel principal de administracion.
- La ruta `GET /admin/entidades` muestra el listado de todas las entidades.
- El listado permite filtrar por estado: `TODAS`, `PENDIENTE`, `VERIFICADA`, `RECHAZADA` y `SUSPENDIDA`.
- Si se recibe un filtro no valido, el listado se trata como `TODAS`.
- La ruta `GET /admin/entidades/:entityId` muestra el detalle de la solicitud.
- Las rutas administrativas requieren sesion activa y rol `ADMIN`.

### 2. Cambio de estado de entidad

- Se ha creado la ruta `POST /admin/entidades/:entityId/estado`.
- El controlador administrativo procesa acciones de `PENDIENTE`, `VERIFICADA`, `RECHAZADA` y `SUSPENDIDA`.
- El servicio `updateEntityValidationStatus` actualiza el estado en base de datos.
- El servicio ya bloquea transiciones invalidas y cambios al mismo estado.
- Si la entidad no existe, la operacion devuelve error de entidad no encontrada.
- Tras un cambio correcto, el administrador vuelve al detalle de la entidad.

### 3. Trazabilidad administrativa

- Se ha aﺣﺎadido el modelo `AdminActionLog` en Prisma.
- Cada cambio de estado genera un log con:
  - accion administrativa
  - estado anterior
  - estado nuevo
  - notas
  - administrador responsable
  - entidad afectada

### 4. Restriccion para publicacion de eventos

- Se ha creado `src/services/entities/entity-publication-access.service.js`.
- Este servicio lanza error si la entidad no esta en estado `VERIFICADA`.
- Una entidad en estado `SUSPENDIDA` no puede operar ni publicar eventos.
- Si una entidad vuelve a `VERIFICADA`, recupera la capacidad de operar siempre que cumpla las reglas actuales del sistema.

### 5. Interfaz

- Se ha creado el listado administrativo en `src/views/pages/admin/entities-list.ejs`.
- Se ha creado la vista de detalle y moderacion en `src/views/pages/admin/entity-detail.ejs`.
- El listado muestra nombre de entidad, solicitante, email, estado, ciudad, fecha de solicitud y enlace al detalle.
- El detalle muestra datos de la entidad, solicitante, estado actual, formulario de cambio y log historico.

### 6. Bootstrap de admin local

- Se ha aﺣﺎadido el script `npm run seed:admin`.
- El script crea o actualiza un administrador local para pruebas.
- El email y la contraseﺣﺎa se leen desde `ADMIN_EMAIL` y `ADMIN_PASSWORD` en `.env`.
- Si falta alguna de esas variables, el script falla con un mensaje claro y no crea credenciales por defecto.

## Comprobacion manual

1. Ejecutar `npm run seed:admin`.
2. Ejecutar `npm run dev`.
3. Iniciar sesion con el email y la contraseﺣﺎa configurados en `ADMIN_EMAIL` y `ADMIN_PASSWORD`.
4. Confirmar la redireccion a `http://localhost:3000/admin/perfil`.
5. Abrir `http://localhost:3000/admin/area` desde el menu.
6. Abrir `http://localhost:3000/admin/entidades`.
7. Probar los filtros `TODAS`, `PENDIENTE`, `VERIFICADA`, `RECHAZADA` y `SUSPENDIDA`.
8. Entrar al detalle de una entidad.
9. Seleccionar `Marcar como pendiente`, `Verificar`, `Rechazar` o `Suspender` segun el estado actual.
10. Guardar el cambio y comprobar la redireccion al detalle.
11. Revisar el historial de acciones en el detalle de la entidad.
